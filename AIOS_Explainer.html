AIOS — how it actually works

A plain-language map of the moving parts: a message's journey, how things get remembered, and how agents do work. No app UI here — just the machine, made legible.

Feeling anxious about silent gaps? Read this first. The thing you're afraid of — quiet, invisible failure — is the single principle this whole system was designed around ("these systems fail silently"). That's why everything is fail-closed, observed, and checked against fixtures. Each of your 46 build issues has a testable "done." You build one layer at a time, and each one is verifiable before the next. You are not flying blind.

🧠Three things, one system

At the highest level AIOS is a brain (memory), a workforce (agents), and a habit of honest answering (provenance) — all kept inside one client's walls.

🧠 The brain
Durable, permission-safe memory of what the business knows — decisions, preferences, lessons, SOPs.
🤝 The workforce
Configurable agents that act on the memory — research, draft, score, orchestrate.
🏷️ Honest answers
Every answer is labelled by where it came from, and the brain abstains rather than make things up.
The one rule that governs everything (the spine): store something in memory only if it's not already a field in a system of record and it has lasting interpretive value. Current facts (a deal stage, an invoice) are fetched live, never copied. On conflict, the system of record always wins.

The three engines that "know what to do"

There is no single magic brain deciding things. It's three understandable engines:

✍️ Write — the routing gates
Decide what comes in, whether it's worth keeping, and as what type. (Tab ③)
🔎 Read — retrieval
Pull the relevant memories, permission-filtered and labelled, or abstain. (Tab ②)
🔧 Maintain — the crons
Consolidate, decay, prune — the brain keeps itself healthy while you sleep. (Tab ④)
Why isolation is calm-making: each client gets their own database + their own engine. One client physically cannot reach another's data — not "filtered from," cannot address. The code is one shared codebase; only the data is separate.

📥Ingestion — the four gates

Everything that comes in (an email, a Slack message, a doc, a meeting transcript, a CRM event) runs these gates in order, cheap → expensive, and stops at the first match. The deterministic, free gates run first so an LLM is only used when truly needed.

1
Sensitive or excluded?
From a do-not-ingest source (HR, legal, #personal) or carrying a sensitivity label → DROP never stored, never indexed. Deterministic · no LLM
2
A current-state structured fact?
Only for structured connectors. Is the field in the schema registry (e.g. deal_stage)? → FETCH LIVE never copy it into memory. Unknown field → review, never auto-store. Deterministic · no LLM, ever
3
Lasting interpretive value?
A cheap classifier first; an LLM only on the ambiguous ones. No / Unsure → INDEX-IN-PLACE (searchable "chunks", no lifecycle). Yes → continue. Safe default = keep it searchable, never auto-promote uncertain content.
4
Also a structured action?
Produces a task/contact/pipeline change? → WRITE BOTH the structured part to the system of record and an episodic memory of where it came from. Otherwise → WRITE TO MEMORY (pick the type — Tab ④).
After any write: attach provenance · sensitivity = the highest of its sources · zone = the union · skip if a content-hash duplicate · invalidate any older fact it replaces. A low-trust source (inbound external email, web) can be made searchable but cannot become a trusted "I know this" memory without corroboration or human review — this stops bad/injected content from laundering itself into fact.
Nothing is invisible: every gate decision (even DROP) is written to an ingestion-decision log (references, not content). That's what lets you later ask "did we wrongly drop something important?" — the one blind spot most systems never close.

How a memory lives and dies

This cycle is the moat. Most "RAG apps" only have the searchable-chunks half — none of this lifecycle.

📥 ingest → episodic → nightly consolidation → semantic → weekly decay tidies up → SOPs are never decayed
The guard against silent loss: episodic is only ever deleted once a confirmed semantic child points back to it. If consolidation ever misses one, that episodic is kept (and flagged), never silently dropped. A coverage metric watches for exactly this.
Two extra filters, always applied: namespace (is this about the right client/project — don't bleed Client A into Client B) and permission (is the asker cleared to see it). Both fail closed: when unsure, show less.

🤝How the right task reaches the right agent

Two routing layers. The first picks answer vs act; the second picks how to act — and most repeatable work uses the predictable, non-magical path.

Mode A · Workflow deterministic

When the steps are known. A JSON recipe runs exactly the same way every time. Cheap, predictable, no surprises.

new CRM contact
└→ researcher
   └→ scorer
      └→ email-writer (if score > 7)
Mode B · Orchestrator dynamic

When the goal is open-ended ("prep me for the Acme meeting"). The orchestrator reads the agent registry and plans who does what.

🧭 Orchestrator — "prep for Acme"
├→ 🔍 Researcher — pull Acme history
├→ 📊 Analyst — summarise the account
└→ ✍️ Writer — draft the brief
    └→ 🔍 sub-agent — look up one detail

What makes routing reliable (not a black box)

MechanismWhat it does
🗂️ Agent registryEach specialist advertises its persona, skills and allowed tools. The orchestrator matches sub-goals to the best-fit agent — like tool-calling, but the "tools" are agents.
🪪 Principal inheritanceThe whole tree acts as the one user/service that triggered it. A sub-agent 3 levels deep can never exceed that person's permissions or borrow someone else's tokens.
📈 Trust scoreAgents that fail or get rejected lose trust → constrained (need approval) → quarantined (disabled). Bad agents don't quietly keep running.
🙋 Clarification interruptA stuck sub-agent doesn't guess or loop — it pauses (durable state), asks via the Inbox, and resumes when answered.
🛑 Confirmation gateAnything irreversible/external (send email, change a record, touch money) previews → confirms before doing it.
🌳 Live delegation treeWho handed what to whom is visible and traced — multi-agent is real and watchable, not implied.
Honest caveat → now addressed: dynamic routing is only as good as the agent descriptions, and deep trees cost more and are harder to debug. So it's specced explicitly: each agent carries a structured capability manifest (whenToUse, capabilities, exampleGoals) the orchestrator routes on; a deterministic pre-filter narrows the candidate set before the planner runs; and delegation depth is capped by orchestrator_max_depth (default 3) to keep trees shallow. Use workflows for known paths, the orchestrator only for open-ended ones.

🛡️Why it won't fail silently — the part for your anxiety

No system is immune to silent failure — pretending otherwise is the trap. The honest goal is to shrink the hidden surface to almost nothing and put a guard on each piece. Here's what's already designed in.

The fearThe guard that's built in
Showing data to the wrong personFail-closed permissions applied in the database before ranking + adversarial leak tests in CI (issue #36). When unsure, it shows nothing.
Confident, wrong answersProvenance per claim + abstain over guess + a verification pass on cited claims. "General inference" is visually marked, never a fact.
Quietly losing knowledgeEpisodic only deleted with a confirmed semantic child; SOPs never decay; invalidate-don't-overwrite keeps full history.
Wrongly dropping something at ingestionIngestion-decision log + sampled audit + miss↔ingestion cross-check — you can see what was dropped.
Quality eroding over weeksQuality Monitor (abstention/miss/utility trends) judged by held-out eval fixtures, not metrics that move themselves.
The monitors themselves dyingWatching the watchers (#45): dead-man's switch on every cron, embedding-drift canary, completeness critic, and a CI test that no model call escapes the gateway.
One client leaking into anotherPhysical isolation: separate DB + engine + queue per client. No shared query path exists to leak through.
Your real safety net while building: every one of the 46 issues has acceptance criteria — a concrete, testable "done." You don't have to hold the whole system in your head. Build the next issue, make its acceptance test pass, move on. The architecture's been pressure-tested three times; what's left is implementation you verify one slice at a time.
Where to start (least scary path): Milestone 0 — the tracer-bullet slice (issues #1–#5): upload one SOP → embed → retrieve → answer with a provenance label → abstain when it doesn't know. That one thread proves the riskiest core end-to-end before any of the bigger machinery exists.
